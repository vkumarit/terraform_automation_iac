# Azure Authentication #

>>> Server-side/backend Alternatives for Terraform Authentication <<<

I. Using Service Principal secrets (tenant_id, subscription_id, client_id, client_secret)

A) For first time/step, also when we exit the session with exported vars and
   starting new session. The typical secure workflow is:

> a) > Create an App Registration (Service Principal) in Azure Portal
  - Register an application in Azure AD.
  - Note the Client ID, Tenant ID.
  - Generate a Client Secret.
  - Assign the Service Principal the necessary roles (e.g., Contributor) 
    on your subscription or resource group.
  - generate Personam Access Token from azure organization to setup agent on vm.
  
  b) > Create Service Principal using Azure AD CLI
  - Log into Azure (Cloud Shell or local machine) and do:
    $ az ad sp create-for-rbac --name "terraform-ec2-sp" --role Contributor \
        --scopes /subscriptions/<SUBSCRIPTION_ID> --sdk-auth
    This prints JSON including clientId (AppId), clientSecret, tenantId, subscriptionId. Store them 
    securely and use on the EC2 instance.
(SKIP IF APP REGISTRATION DONE)
  * For key vault access assign 'get' access to this service principal when needed to fetch secrets from      key vault

## Azure authentication from outside Azure (like terraform on EC2) ##

> b) > Export the Service Principal credentials as environment variables on our EC2 instance, do bash
  $ export ARM_CLIENT_ID="<client-id-from-app-registration>"
  $ export ARM_CLIENT_SECRET="<client-secret-from-app-registration>"
  $ export ARM_SUBSCRIPTION_ID="<our-subscription-id>"
  $ export ARM_TENANT_ID="<tenant-id-from-app-registration>"
  This enables Terraform to authenticate securely without embedding secrets in code.
  Terraform will read the ARM_* env vars and authenticate, can use below provider block at minimum:
	provider "azurerm" {
	  features {}
	}

  Change the exported env vars as
    1) To see all env vars,
       $ env or printenv
    2) filter by pattern
       $ printenv | grep ARM_* 
    3) To check single variable,
       $ echo $ARM_CLIENT_ID
    4) To reset a variable export it again as
       $ export ARM_CLIENT_ID="new-client-id"
    5) Remove from current shell,
       $ unset ARM_CLIENT_SECRET

  Also used in dev, test environment.

or,

  b) > Azure CLI Interactive or Service Principal Login:
  - Run 'az login' (interactive or with SP credentials) on your EC2 instance before running Terraform.
    $ az login \
        --service-principal \
        --username <clientId> \
        --password <clientSecret> \
        --tenant <tenantId>
    
    If getting error 'No subscriptions found for <client-Id>.`, go to subscription > IAM > 
    Add role assignment > Role > Priviledged admin. roles > select 'Owner'/'contributor' > Members > User,         group, or service principal > Select members > 'az-classic-app' (name of app registered)> assigns role 
    'owner' at scope 'This resource'.

  - Verify the subscription used by terraform.
    $ az account show
    
  - Configure Terraform to use AzureRM provider
  
    Terraform >=4 requires subscription_id for azure authentication, so
    i) Export subscription id to env vars,
       $ export ARM_SUBSCRIPTION_ID="<our-subscription-id>" 
       
       terraform {
         required_providers {
           azurerm = {
             source  = "hashicorp/azurerm"
             version = "~> 4.0"
          }
         }
       }
    
       provider "azurerm" {
         features {}
       }
    
    ii) Hardcode subscription id as below without exporting to env vars,
       provider "azurerm" {
         features         {}
         subscription_id  = "<-subscription_id->"
       }
    
    iii) Else, after exporting id to env vars, explicitly reference as a variable
       variable "subscription_id" {
          description = "Azure subscription ID"
          type        = string
          default     = null  # Falls back to env var
       }

       provider "azurerm" {
         features         {}
         subscription_id  = var.subscription_id
       }

    

    When the Azure CLI is logged in with a user? or service principal, AzureRM can automatically use the SP     credentials.

  - Terraform picks up the authentication context from the Azure CLI session.
  Use Case: Manual runs or environments where you can run CLI commands. Can also be used subsequently   afterwards initial setup.

> c) > Run terraform init, plan -out ('-out' so terraform guarantee to act exactly as was shown in plan phase),
       and apply commands EC2 instance. Terraform will use these environment variables (or az login context)

V. CI/CD Pipeline Secrets Storage  
  - Use the built-in secrets management features of CI/CD tools 
  - such as GitHub Actions, Azure DevOps, or Jenkins
  - store and encrypt secrets then inject them as environment variables into build and deployment jobs.
  - Env - prod.

 > How to configure CI/CD or automation agents to inject Service Principal secrets as environment 
   variables securely for terraform?
  
  a) With Azure DevOps: 
     In classic pipelines, environment variables mapped in one task do not persist to other tasks.
       1) Secret Variables:
          > Go to Pipelines > Select pipeline > Edit.
          > Click on the Variables tab.
          > Add a new variable (e.g., ARM_CLIENT_SECRET, ARM_CLIENT_ID, ARM_TENANT_ID, ARM_SUBSCRIPTION_ID).
          > Enter value > check Keep this value secret.
          > Save pipeline.
          > Classic Pipeline - Go to Environment Variables, map the secret variable to an environment 
            variable (e.g., variable name as ARM_CLIENT_SECRET and value as $(ARM_CLIENT_SECRET), 
            so Terraform on vm picks them up as ARM_* variables during that tasks' runtime. The pipeline tasks             we do this with are below, 
            i) a Bash or PowerShell script task, in a script body add below and run it,
               
               terraform init
               terraform plan -out=tfplan
               terraform apply tfplan
 
            ii) a Terraform task (by microsoft), configured with Environment Variables as mentioned. 

          > YAML Pipeline - In YAML file, reference the variable as $(ARM_CLIENT_SECRET). 
            To use it in a script, map it to an environment variable as:

            steps:
            - script: |
                echo "Secret value is set"
                # Terraform steps here
              env:
                ARM_CLIENT_SECRET: $(ARM_CLIENT_SECRET)

​
       2) Variable Groups (recommended for reuse, used in production):
          > Go to Azure DevOps project > Pipelines > Library > Click + Variable group.
          > Enter a name for the group (e.g., prod-terraform-vars). 
          > Add variables (e.g., ARM_CLIENT_SECRET, ARM_CLIENT_ID, ARM_TENANT_ID, ARM_SUBSCRIPTION_ID) and 
            check the secret checkbox for sensitive values.
          > Save the variable group.
          > Classic Pipeline - Edit pipeline > Variables tab > Click Link variable group 
            > select prod-terraform-vars group > Save pipeline. 
              In pipeline tasks, under Environment Variables, map the secret variable to an                                   environment variable (e.g., var as ARM_CLIENT_SECRET and value as $(ARM_CLIENT_SECRET),
              so Terraform picks them up as ARM_* variables during that tasks' runtime, The pipeline tasks                    we do this with are below,
              i) a Bash or PowerShell script task, in a script body add below and run it,
                
                terraform init
                terraform plan -out=tfplan

          Recommended for production - Store service principal secrets (client secret) and other sensitive                values in Azure Key Vault. Expose them to Azure DevOps pipeline via a variable group linked to 
          that Key Vault.
          i) Store secrets in Azure Key Vault: 
             a) Azure portal > Create/choose Key Vault in prod env, and

                The actual values stay only in Key Vault, Azure DevOps just references them.
          iii) Connect variable group to the production pipeline -
             a) Open production classic pipeline > click Edit > Variables tab > click “Link variable group”

       4)  Use a task such as AzureCLI@2 or the Terraform task -

           ability to authenticate and manage Azure resources, and it's the connection used by

           Azure authentication.
           i) Using AzureCLI@2 - 
                a) Install azure cli on ec2 instance once, manual az login is not required.
                b) Create task AzureCLI@2 in classic pipeline and select ARM service connection (SP)

                     Or,
                   - 'Use Entra ID for Authentication' option is when we want to authenticate using Azure AD                         (Entra ID) credentials, such as managed identities or user-based authentication.

How to use `terraform import` in both **Azure** and **AWS** contexts, either from a **CI/CD pipeline** or **locally**. 

Step-by-step:

1. Understanding `terraform import`

* `terraform import` is used to 'bring existing infrastructure under Terraform management'.
* It DOESN'T generate `.tf` configuration automatically; we must write the resource block yourself.
* It **links an existing resource** in your cloud account to a Terraform resource in your state file.

Syntax:
$ terraform import [options] <RESOURCE_ADDRESS> <RESOURCE_ID>
* `RESOURCE_ADDRESS` → The Terraform resource name in your `.tf` files, e.g., `aws_s3_bucket.my_bucket`.
* `RESOURCE_ID` → The actual ID of the resource in the cloud provider.

2. Using `terraform import` Locally

### **Step-by-step (AWS/Azure example)**

- Create a `.tf` resource block (without applying yet):

resource "aws_s3_bucket" "my_bucket" {
  bucket = "my-existing-bucket"
}

resource "azurerm_resource_group" "my_rg" {
  name     = "existing-rg"
  location = "East US"
}

- Initialize Terraform:
$ terraform init

- Import the resource:
$ terraform import aws_s3_bucket.my_bucket my-existing-bucket
$ terraform import azurerm_resource_group.my_rg /subscriptions/<SUBSCRIPTION_ID>/resourceGroups/existing-rg


- Check state:
$ terraform state list
$ terraform state show azurerm_resource_group.my_rg

- Run `terraform plan` > differences > update `.tf` file > match actual resource attributes.

3. Using `terraform import` in a Pipeline (CI/CD)

If running Terraform in a pipeline (GitHub Actions, Azure DevOps, Jenkins, etc.), we need to **authenticate** with cloud provider and ensure Terraform can write to the state backend.

### **Pipeline Steps Example (AWS)**

```yaml
steps:
  - name: Checkout code
    uses: actions/checkout@v3

  - name: Setup Terraform
    uses: hashicorp/setup-terraform@v2

  - name: Configure AWS credentials
    uses: aws-actions/configure-aws-credentials@v2
    with:
      aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }}
      aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
      aws-region: us-east-1

  - name: Terraform Init
    run: terraform init

  - name: Terraform Import
    run: terraform import aws_s3_bucket.my_bucket my-existing-bucket

  - name: Terraform Plan
    run: terraform plan
```

4. **Summary**

* `terraform import` is for linking existing resources to Terraform.
* Syntax is provider-specific but conceptually the same.
* `Write the resource block before importing.` Terraform won’t create `.tf` files for you.
* Check resource IDs carefully.** AWS uses names/ARNs; Azure uses full resource IDs.
* Locally: run `terraform init` > `terraform import` > `terraform plan`.
* In pipelines: authenticate > init > import > plan > apply (optional).
* `Use remote state in pipelines` to prevent state conflicts and also if want the state shared across runs.
* Always run `terraform plan` after import** to sync your config with real resource settings.
* Always verify and adjust the `.tf` config after import.
* Automate carefully** — importing is usually a `one-time operation` per resource, not a repeated pipeline step.

*** Handling in Production (IMPORTANT) ***

## Step-by-step workflow - To bring the manually created resource under Terraform control SAFELY.

-> Step 1: Identify the resource details
From dashboard / CLI, find out Name, ID, Region, Resource group / VPC / subnet, Tags. 
Example: AWS S3 bucket: `prod-logs-bucket`, Azure RG: `prod-rg-extra`.

-> Step 2: Add resource block to `.tf` (NO APPLY) - Examples AWS/Azure:

resource "aws_s3_bucket" "prod_logs" {
  bucket = "prod-logs-bucket"
}

resource "azurerm_resource_group" "prod_extra" {
  name     = "prod-rg-extra"
  location = "East US"
}

-> Step 3: Initialize (uses remote backend)
$ terraform init
Terraform connects to S3 / Azure Blob where Existing production state is located.

-> Step 4: Import the resource into remote state (KEY STEP). AWS/Azure examples,
$ terraform import aws_s3_bucket.prod_logs prod-logs-bucket
$ terraform import azurerm_resource_group.prod_extra \
/subscriptions/<SUB_ID>/resourceGroups/prod-rg-extra
Resource is now written into **remote state**. Safe for pipelines and team.

-> Step 5: Inspect imported state AWS/Azure
$ terraform state show aws_s3_bucket.prod_logs
$ terraform state show azurerm_resource_group.prod_extra
This shows the **real config** from cloud.

-> Step 6: Reconcile `.tf` with reality. Now running plan, shows '~ update in-place'.
$ terraform plan

Reason - Dashboard defaults ≠ Terraform defaults, Missing attributes, Tags differ, Encryption flags, policies, etc. Now fix `.tf` file until it shows 'No changes. Infrastructure is up-to-date.'.

-> Step 7: Commit and let pipeline run
$ terraform plan
$ terraform apply
The imported resource is **fully Terraform-managed**.

## Rotate secrets regularly and enable alerting

A) Rotation:
	1) Manually: Periodically update the secret value in Key Vault (new SP secret/password, DB password, 		
    etc.). Each update creates a new secret version; update the downstream system (e.g., SP or DB) to use the new value.
	2) Automated: Use: 
		a) Event Grid + Azure Function / Logic App that listens for “SecretNearExpiry” events and 			
        rotates the secret (creates new value, writes to Key Vault, updates the dependent resource).
		b) For keys, use key rotation policies configured on Key Vault keys.
​
B) Alerting:
	1) Use Azure Monitor / Alerts on:
		a) Key Vault logs (audit log, secret access, secret change events).
		b) “SecretNearExpiry” events, unauthorized access attempts, or abnormal access patterns.
	2) Route alerts to email, Teams, or incident tools.

​> Secret Rotation: When rotating secrets, update both Key Vault and credentials used by automation/bots 
running Terraform.

Terraform Pipeline Failure Recovery Architecture / Terraform Failure-Resilient Deployment Pipeline
==================================================================================================
Overview

This Terraform pipeline is designed to ensure reliable infrastructure provisioning even when a Terraform apply 
operation fails. The pipeline includes mechanisms to detect and clean partially created Terraform resources, 
resources created in Azure but not recorded in Terraform state, and broken infrastructure caused by failed provisioning.

The main goal is to ensure that every pipeline run produces deterministic infrastructure based on Terraform code, 
without requiring manual cleanup.

---

High Level Pipeline Flow
^^^^^^^^^^^^^^^^^^^^^^^^

Terraform Init
↓
Terraform Plan
↓
Terraform Apply
      │
      ├─ SUCCESS → Pipeline ends
      │
      └─ FAILURE
            ↓
      terraform apply -refresh-only
            ↓
      detect partially created Terraform resources
            ↓
      destroy broken resources + remove from Terraform state
            ↓
      scan Azure for orphan resources (tag: terraform_run)
            ↓
      delete orphan resources
            ↓
      pipeline ends
            ↓
      next pipeline run recreates infrastructure cleanly
---
Goal
----

Ensure Terraform deployments remain deterministic and self-healing even when
a `terraform apply` fails.

The pipeline automatically:

• Detects partially created resources
• Synchronizes Terraform state with Azure infrastructure
• Cleans broken infrastructure
• Detects Azure orphan resources
• Deletes unmanaged resources
• Ensures the next pipeline run starts from a clean environment


--------------------------------------------------------------------

Normal Deployment Flow
----------------------

Step 1 – Terraform Init
-----------------------

Terraform initialization prepares the working directory for Terraform operations.

Responsibilities of `terraform init`:

• Download required provider plugins
• Initialize backend configuration
• Configure remote state storage
• Download Terraform modules
• Prepare the working directory

Example command:

terraform init -reconfigure -lock-timeout=5m

What happens internally:

1. Terraform reads backend configuration.
2. It connects to remote state storage.
3. Provider plugins (such as AzureRM) are downloaded.
4. Required modules are downloaded.
5. Terraform prepares the working directory.

Result:

Terraform is ready to communicate with Azure and the remote state backend.


--------------------------------------------------------------------

Step 2 – Terraform Plan
-----------------------

Terraform plan determines what infrastructure changes are required.

Responsibilities:

• Compare Terraform configuration with Terraform state
• Compare Terraform state with real Azure infrastructure
• Determine required resource changes

Example command:

terraform plan \
-input=false \
-parallelism=5 \
-lock-timeout=10m \
-no-color \
-out=tfplan.binary

Terraform compares three things:

1. Terraform configuration (your .tf files)
2. Terraform state (stored in backend)
3. Actual Azure infrastructure

Plan results:

Exit code 0 → No changes  
Exit code 2 → Changes detected  
Exit code 1 → Error

The plan is saved as a binary file:

tfplan.binary

Purpose of saving plan:

Ensures the exact reviewed plan is executed during apply.


--------------------------------------------------------------------

Step 3 – Terraform Apply
------------------------

Terraform apply executes the plan generated in the previous step.

Example command:

terraform apply \
-input=false \
-auto-approve \
-parallelism=5 \
-lock-timeout=10m \
tfplan.binary

Terraform performs:

• Resource creation
• Resource modification
• Resource deletion (if needed)
• Dependency graph execution
• Terraform state updates

If everything succeeds:

• Terraform state is updated
• Azure infrastructure matches Terraform configuration
• Pipeline ends successfully


--------------------------------------------------------------------

Failure Scenario
----------------

Failures can occur during the apply stage.

Common causes:

• Marketplace image terms not accepted
• VM provisioning failures
• Azure quota limits
• Insufficient permissions
• Azure API failures
• Network interruptions
• Provider crashes

Example failure scenario:

1. Network interface created
2. Disk created
3. Virtual machine created
4. OS provisioning fails
5. Terraform apply stops

Result:

Partially created infrastructure remains in Azure.


--------------------------------------------------------------------

Failure Recovery Flow
---------------------

If `terraform apply` fails, the pipeline enters recovery mode.


--------------------------------------------------------------------

Step 4 – Refresh Terraform State
--------------------------------

After a failure the pipeline runs:

terraform apply \
-refresh-only \
-auto-approve \
-lock-timeout=10m

Purpose:

Synchronize Terraform state with the real Azure infrastructure.

Important behavior:

• Does NOT create resources
• Does NOT destroy resources
• Only updates Terraform state

What this fixes:

If Terraform created resources before failure but never recorded them in state,
refresh-only detects them and updates the state file.


--------------------------------------------------------------------

Step 5 – Generate Recovery Plan
-------------------------------

Purpose:

Detect partially created or broken Terraform resources.

Commands:

terraform plan \
-input=false \
-parallelism=5 \
-lock-timeout=10m \
-out=tfplan.recovery

terraform show -json tfplan.recovery > tfplan.recovery.json

The JSON output is analyzed to detect broken resources.


Indicators of broken resources:

• Resource scheduled for creation again
• Incomplete provisioning state
• Missing attributes
• Provider failure during creation


These resources are considered partially created Terraform resources.


--------------------------------------------------------------------

Step 6 – Destroy Broken Terraform Resources
-------------------------------------------

Broken resources must be removed so Terraform can recreate them correctly.

Destroy resource in Azure:

terraform destroy \
-target=<resource_address> \
-auto-approve \
-parallelism=5

Example:

terraform destroy -target=azurerm_linux_virtual_machine.vm


Purpose:

Remove partially created infrastructure from Azure.


--------------------------------------------------------------------

Step 7 – Remove Broken Resources From Terraform State
-----------------------------------------------------

Remove failed resources from Terraform state.

Command:

terraform state rm <resource_address>

Example:

terraform state rm azurerm_linux_virtual_machine.vm

Result:

Terraform will treat the resource as new during the next run.


--------------------------------------------------------------------

Step 8 – Detect Azure Orphan Resources
--------------------------------------

Sometimes Azure resources exist but Terraform never records them in state.

Possible causes:

• Terraform provider crash
• Network interruption
• State update failure
• Pipeline crash
• Terraform process killed

These resources are called orphan resources.


To detect them, all Terraform resources are tagged with a pipeline run identifier.

Example tag:

terraform_run = <pipeline_run_id>

Example:

terraform_run = 4521


Azure Resource Graph query detects resources with this tag.

Command:

az graph query -q "
Resources
| where tags.terraform_run == '<run_id>'
| project id
" --query "data[].id" -o tsv


This query returns all resources created during the failed run.


--------------------------------------------------------------------

Step 9 – Delete Orphan Azure Resources
--------------------------------------

If a resource:

• Has the terraform_run tag
• Does not exist in Terraform state

Then it is considered orphan infrastructure.

Delete command:

az resource delete --ids <resource_id>

Example:

az resource delete --ids \
/subscriptions/xxxx/resourceGroups/rg/providers/Microsoft.Compute/virtualMachines/vm1


Purpose:

Ensure no unmanaged infrastructure remains in Azure.


--------------------------------------------------------------------

Step 10 – Pipeline Ends After Cleanup
-------------------------------------

After cleanup completes:

• Terraform state becomes consistent
• Azure infrastructure is cleaned
• Broken resources removed
• Orphan resources deleted

The pipeline stops.

Automatic retry is avoided because failures may require
code changes or configuration fixes.


--------------------------------------------------------------------

Step 11 – Next Pipeline Run
---------------------------

After fixing configuration issues or external problems,
the pipeline runs again.

Execution flow:

terraform init -reconfigure

terraform plan -out=tfplan.binary

terraform apply tfplan.binary


Since broken and orphan resources were removed earlier,
Terraform detects missing infrastructure and recreates it cleanly.


--------------------------------------------------------------------

Key Design Principles
---------------------

Deterministic Infrastructure

Every pipeline run must produce the same infrastructure
from the Terraform code.

Manual infrastructure changes are not trusted.


Automatic Cleanup

Partially created resources can break dependency graphs.
Automated cleanup prevents cascading failures.


Accurate Terraform State

Terraform state must always match the real infrastructure.
State inconsistencies must be repaired before the next run.


No Unmanaged Infrastructure

Azure resources must never remain unmanaged.
Tag-based detection ensures cleanup.


--------------------------------------------------------------------

Advantages of This Architecture
-------------------------------

Reliable infrastructure provisioning

Even if provisioning fails, the system cleans itself
before the next run.


Automated recovery

Most failures are handled automatically
without manual intervention.


Drift prevention

Azure infrastructure always matches Terraform configuration.


Safe CI/CD infrastructure deployment

The pipeline becomes resilient to common Terraform failures.


--------------------------------------------------------------------

Summary
-------

This pipeline implements a failure-resilient Terraform deployment workflow.

Key capabilities:

• Detect partially created Terraform resources
• Synchronize Terraform state with Azure infrastructure
• Remove broken infrastructure
• Detect orphan Azure resources
• Delete unmanaged resources
• Ensure clean deployments on next run


By combining Terraform state reconciliation with Azure
resource scanning, the pipeline guarantees that every
deployment begins from a clean and consistent
infrastructure state.

Terraform Failure Classes and How the Pipeline Handles Them

Goal

Terraform deployments can fail in multiple ways. A robust CI/CD pipeline must detect each failure type and 
recover automatically where possible.

This document describes six common Terraform failure scenarios and the recovery mechanism implemented in the
pipeline.

---

Failure Type 1 – Terraform Apply Failure Before Resource Creation

Example causes:

Invalid Terraform configuration
Syntax errors
Provider validation failure
Invalid resource parameters

Example:

Invalid VM size
Invalid subnet reference
Wrong variable value

What happens:

Terraform stops before creating any infrastructure.

Infrastructure state:

No Azure resources created
Terraform state remains unchanged

Pipeline behavior:

Terraform apply fails
Pipeline exits
No recovery steps needed

Next run:

Fix configuration and run pipeline again.

---

Failure Type 2 – Partial Resource Creation

Example scenario:

Terraform creates some resources but fails before finishing.

Example:

NIC created
Disk created
VM created
OS provisioning fails

Infrastructure state:

Some resources exist in Azure
Terraform state may contain incomplete data

Pipeline recovery process:

Run terraform apply -refresh-only

This synchronizes Terraform state with Azure infrastructure.

Then run recovery plan and detect partially created resources.

Broken resources are destroyed:

terraform destroy -target=<resource>

Then removed from state:

terraform state rm <resource>

Next pipeline run recreates them cleanly.

---

Failure Type 3 – Azure Resource Created but Not Recorded in Terraform State

Example causes:

Provider crash
Network interruption during state update
Pipeline crash after resource creation

Example:

Azure VM created successfully
Terraform crashes before writing state

Infrastructure state:

Azure resource exists
Terraform state does not contain the resource

This is called an orphan resource.

Pipeline recovery process:

All Terraform resources include a tag:

terraform_run = <pipeline_run_id>

Pipeline scans Azure using Azure Resource Graph.

Example query:

Resources
| where tags.terraform_run == '<run_id>'

If resource is found but not present in Terraform state:

Delete resource using:

az resource delete --ids <resource_id>

Next pipeline run recreates the resource normally.

---

Failure Type 4 – Terraform State Drift

Example scenario:

Infrastructure is modified outside Terraform.

Examples:

Manual Azure portal changes
Manual CLI modifications
Accidental resource updates

Infrastructure state:

Azure infrastructure differs from Terraform state.

Pipeline behavior:

Terraform plan detects drift.

Example plan result:

Resource update required
Resource replacement required

Terraform apply reconciles infrastructure with Terraform configuration.

---

Failure Type 5 – Dependency Graph Failure

Example scenario:

Terraform dependency chain breaks during apply.

Example:

VM depends on NIC
NIC depends on subnet

If subnet creation fails:

NIC fails
VM fails

Infrastructure state:

Some dependency resources may exist
Other resources fail to create

Pipeline recovery:

Partial resources detected during recovery plan.

Broken resources are destroyed and removed from state.

Next pipeline run rebuilds dependency chain correctly.

---

Failure Type 6 – Azure API or Provisioning Failures

Example causes:

Azure service outage
Quota exceeded
Region capacity issues
Marketplace image agreement not accepted

Example:

VM provisioning fails after creation request.

Infrastructure state:

Some resources may exist but remain unusable.

Pipeline recovery:

Refresh Terraform state

Detect failed resources using recovery plan

Destroy broken resources

Remove them from Terraform state

Scan Azure for orphan resources

Delete them

Next pipeline run recreates infrastructure after issue is fixed.

---

Pipeline Self-Healing Architecture

Normal flow:

Terraform Init
Terraform Plan
Terraform Apply

If apply succeeds:

Pipeline completes successfully.

If apply fails:

Terraform refresh-only
Detect partially created resources
Destroy broken resources
Remove resources from Terraform state
Scan Azure for orphan resources using terraform_run tag
Delete orphan resources

Pipeline ends.

Next pipeline run recreates infrastructure cleanly.

---

Benefits of This Architecture

Handles partial infrastructure failures automatically
Prevents infrastructure drift
Prevents orphan Azure resources
Maintains consistent Terraform state
Ensures deterministic infrastructure deployments

---

End Result

The pipeline becomes a self-healing infrastructure system that automatically recovers from common Terraform 
deployment failures and ensures that infrastructure always matches the Terraform configuration.

Terraform also caches provider plugins in a global CLI plugin cache (for example ~/.terraform.d/plugin-cache 
or ~/.terraform.d/plugins). On self-hosted agents, this cache persists across pipeline runs. That means even
if .terraform in the working directory is deleted, Terraform may reuse an older cached provider from the 
global cache. For example, Run 1 downloads azurerm v3.x into the cache; in Run 2 Terraform reuses that cached
version, whose schema does not support the default_tags block, resulting in the error 
“Unsupported block type: default_tags” even though the local .terraform directory was removed.

Running terraform init -upgrade fixes this because Terraform ignores cached provider selections and 
re-evaluates the version constraints defined in the configuration. It then downloads the correct version
(for example azurerm v4.1.0), which supports the default_tags feature, allowing the configuration to work
correctly.

### Terraform Tagging Strategy – Final Implementation Notes (Updated)

#### 🎯 Objective

Implement a consistent tagging strategy in Terraform to:

* Track resources created per pipeline run
* Detect and clean up partially created and orphaned resources
* Standardize metadata across all Azure resources
* Ensure safe recovery from failed Terraform executions
* Prevent re-tagging of existing resources on every run

---

### 🧠 Core Design

We introduced a centralized tagging mechanism using `locals`:

```hcl
locals {
  common_tags = {
    environment = var.environment
    managed_by  = "terraform"
  }
}
```

#### Purpose of each tag:

* **environment** → identifies environment (dev/test/prod)
* **managed_by** → ensures only Terraform-managed resources are targeted
* **creation_run_id** → uniquely identifies the pipeline run that CREATED the resource

---

### ⚠️ Important Design Decision

`creation_run_id` is **NOT part of `common_tags` intentionally**

It is applied per-resource like:

```hcl
tags = merge(local.common_tags, {
  Name            = "<resource-name>"
  creation_run_id = var.run_id
})
```

Reason:

* Allows lifecycle control per resource
* Prevents unwanted updates across runs
* Ensures immutability after creation

---

### 🛠️ How Tagging is Applied

We use Terraform’s `merge()` function to combine:

* global/common tags
* resource-specific tags

```hcl
tags = merge(local.common_tags, {
  Name            = "<resource-name>"
  creation_run_id = var.run_id
})
```

#### Why `merge()`?

* Prevents duplication
* Allows per-resource customization
* Keeps tagging consistent across all resources

---

### 🔒 Preventing Tag Updates (CRITICAL)

To stop Terraform from updating tags every run:

```hcl
lifecycle {
  ignore_changes = [
    tags["creation_run_id"]
  ]
}
```

#### Behavior:

* First run:

  * Resource created
  * `creation_run_id = RUN_ID_1`

* Next run:

  * `var.run_id = RUN_ID_2`
  * Terraform ignores change

✔ Tag remains unchanged
✔ No unnecessary updates
✔ Full idempotency maintained

---

### 🛠️ Extended Lifecycle Usage

For resources with dynamic attributes:

```hcl
lifecycle {
  ignore_changes = [
    customer_managed_key,
    tags["creation_run_id"]
  ]
}
```

✔ Valid syntax
✔ Supports multiple ignored attributes

---

### ❌ Why `terraform_run` Was Removed

Previous approach:

```hcl
terraform_run = var.run_id
```

#### Problem:

* Changes every pipeline run
* Forces Terraform to update ALL resources
* Breaks idempotency
* Causes unnecessary churn

#### Fix:

```hcl
creation_run_id = var.run_id
```

* lifecycle ignore → freezes value after creation

---

### 🔄 Handling Existing Resources

#### Case 1: Resource exists in Terraform state

* Tags added once (in-place update)
* No recreation
* Future runs → no changes due to lifecycle

#### Case 2: Resource exists but NOT in state (orphan)

* Terraform cannot manage or tag it
* Must be handled via:

  * Cleanup script (preferred)
  * Azure CLI
  * Terraform import

---

### 🔗 Pipeline Integration

Pipeline provides:

```yaml

TF_VAR_run_id      = $(Build.BuildId)
RUN_ID             = $(Build.BuildId)
```

#### Usage:

* `TF_VAR_run_id` → used in Terraform tagging
* `RUN_ID` → used in cleanup script

### 🔥 Integration with Cleanup Logic (CRITICAL)

Tagging directly powers orphan cleanup in `runterraform.sh`

#### Step 1: Identify resources from failed run

```bash
Resources
| where resourceGroup == 'myTFResourceGroup'
| where tags.managed_by == 'terraform'
| where tags.creation_run_id == RUN_ID
| project id, type
```

✔ Finds only resources created in the FAILED run
✔ Ignores older stable resources

### Step 2: Compare with Terraform State

* Extract IDs from Terraform state
* Compare with Azure resources

#### Logic:

* In Azure + NOT in state → ORPHAN
* In Azure + in state → VALID

---

### Step 3: Safe Deletion

Delete ONLY resources:

* With `creation_run_id == CURRENT_RUN_ID`
* AND NOT present in Terraform state

✔ Prevents accidental deletion
✔ Ensures safe cleanup

---

### Step 4: Dependency-Aware Deletion

Order:

* virtualMachines
* networkInterfaces
* disks
* publicIPAddresses

✔ Avoids dependency failures

### Step 5: Untagged Safety Net

Scan entire resource group:

If resource:

* NOT in Terraform state
* NOT already deleted

* /swapfile - This is the path to your swap file
* none - Swap does not get mounted to a directory (unlike disks), So none is used instead of a mount point
* swap - Filesystem type = swap, Tells Linux this entry is for swap space
* sw - Mount options, sw = enable swap with default options
* 0 - Dump field, Swap does not need filesystem backups so 0
* 0 - fsck order, Swap should not be checked at boot so 0
* /etc/fstab = filesystem table, tells Linux to mount disks/filesystems/swap automatically at boot.

$ free -h

## Root Cause

The Service Principal secret stored in Key Vault contained a **trailing newline character (`\n`)**.

This happened because:

* Secret was passed into Terraform during creation
* Input contained a newline (likely from copy/paste or file read)
* Terraform stored the value **exactly as-is**
* Key Vault preserved it exactly

Azure authentication requires an **exact match** of the client secret.

Actual value used by pipeline:

```id="val1"
"actual-secret\n"
```

Expected:

```id="val2"
"actual-secret"
```

> Even one extra character = authentication failure

value = trimspace(var.arm_client_secret)

BUILD_SOURCESDIRECTORY: $(Build.SourcesDirectory) 
          
# a predefined system variable,
# The absolute path on the agent where your repo is checked out,
# resolves to example - `/opt/agent/_work/3/s` when `- checkout: self`

Real difference (important)
1. Build.SourcesDirectory (source-focused)

👉 Strictly tied to code checkout

Always points to the repo root
Only meaningful after checkout: self
Represents your Git content location

Think:

“Where is my code?”

- You need repo files, git, paths.

2. System.DefaultWorkingDirectory (execution-focused)

👉 Used by pipeline engine as:

Default directory for script, task, etc.
Where commands run if you don’t specify workingDirectory

Think:

“Where will my commands execute by default?”

- You just need a working folder.

  path: mycustompath
